## 🎯 Что это

Дополнение к боту RemnaShop, которое добавляет два больших модуля:

| Модуль | Стек | Назначение |
|--------|------|------------|
| **Веб-кабинет** | React + Vite + Tailwind | Личный кабинет пользователя в браузере и как Telegram Mini App |
| **Админка / API** | FastAPI | Управление сервисом из веба — всё, что есть в боте, и больше |

Базовый функционал бота берётся из официального образа
`ghcr.io/snoups/remnashop`. Этот репозиторий **накладывает** сверху свой код —
конфигурацию и данные бота он не меняет.

## 🏗 Как устроено

```
┌────────────────────────────────────────────────────────────┐
│  ghcr.io/snoups/remnashop  (базовый образ бота)              │
│        ▲                                                     │
│        │  Dockerfile накладывает admin_src/src/ → /opt/.../src│
│  ┌─────┴───────┐   ┌──────────────┐   ┌────────────────────┐ │
│  │  remnashop  │   │ taskiq worker│   │  taskiq scheduler  │ │
│  │  бот + API  │   └──────────────┘   └────────────────────┘ │
│  │  :5000      │                                             │
│  └─────────────┘   ┌──────────────┐   ┌────────────────────┐ │
│                    │ PostgreSQL   │   │  Valkey (Redis)    │ │
│                    └──────────────┘   └────────────────────┘ │
│  ┌──────────────────────────┐                                │
│  │  cabinet (nginx + React) │  :5002                         │
│  └──────────────────────────┘                                │
└────────────────────────────────────────────────────────────┘
        все сервисы — в docker-сети remnawave-network
```

- **`Dockerfile`** — кладёт `admin_src/src/` поверх образа бота (расширенный API + меню).
- **`cabinet/`** — отдельный multi-stage build (Vite собирает SPA → отдаётся nginx).
- **`docker-compose.yml`** + **`cabinet/docker-compose.cabinet.yml`** — весь стек.

## 📦 Требования

- **Docker** и **Docker Compose** ([как установить](https://docs.docker.com/engine/install/))
- **Уже настроенный и работающий бот RemnaShop** (его `.env` заполнен)
- `openssl` (есть почти везде) — для генерации секретов
- *Только для сценария Б (отдельный сервер):* приватный канал между серверами
  (WireGuard/VPN) и свободный `443` на сайт-сервере для TLS

## 🚀 Установка

Есть **два сценария** — выбери свой:

| Сценарий | Команда | Когда |
|----------|---------|-------|
| **А. Вместе с ботом** (co-located) | `./install.sh` | кабинет на том же сервере, что и бот — обычный случай |
| **Б. На отдельном сервере** | `./install.sh site` | кабинет на отдельной машине от бота |

В обоих случаях сначала клонируем репозиторий:

```bash
git clone https://github.com/alexdsndr161rus2015-maker/remnashop-cabinet.git
cd remnashop-cabinet
```

---

### 🅰️ Сценарий А — на сервере с ботом

**Что нужно заранее:** уже настроенный бот RemnaShop (его `.env` заполнен и бот
работает). Файлы этого репозитория должны лежать **в каталоге бота** — рядом с
его `.env` и `docker-compose.yml`.

```bash
./install.sh
```

Установщик:
1. ✅ проверит зависимости и наличие `.env` бота;
2. ❓ спросит **2 значения**: username бота и URL кабинета (+ опционально email);
3. 🔐 сгенерирует секреты web-части (`APP_API_KEY`, `APP_JWT_SECRET`) сам;
4. ➕ допишет только недостающие переменные — конфигурацию бота не тронет;
5. 🐳 соберёт образы и поднимет весь стек (бот + воркеры + кабинет).

> Повторный запуск безопасен — заполненные переменные остаются как есть.

**После установки** опубликуй наружу **только кабинет** (порт `5002`) — бот/API
уже выведен твоим существующим reverse-proxy. Добавь блок с TLS:

```caddy
cabinet.example.com {
    reverse_proxy 127.0.0.1:5002
}
```

### 🅱️ Сценарий Б — на отдельном сервере

Кабинет на отдельной машине от бота. Связь бот↔кабинет идёт по **приватному
каналу** (WireGuard/VPN), наружу торчит только домен кабинета. Браузер всегда
ходит только на домен кабинета, nginx проксирует `/api/` внутрь по туннелю
(same-origin) — куки и CORS не ломаются.

Нужно сделать **три вещи**: открыть API бота в приватную сеть → поднять туннель
→ поставить кабинет.

#### Шаг 1 — на сервере БОТА: открыть API в приватную сеть

По умолчанию API бота слушает только `127.0.0.1` (локально). Чтобы его увидел
сайт-сервер через туннель, в `.env` бота задай приватный IP бота и перезапусти:

```bash
echo 'API_BIND_HOST=10.8.0.1' >> .env      # ← приватный IP бота (напр. WG-адрес)
docker compose -f docker-compose.yml -f cabinet/docker-compose.cabinet.yml up -d remnashop
```

> Это безопасно для вебхуков: reverse-proxy бота (Caddy/nginx) обычно ходит к
> боту по docker-сети (имя `remnashop:5000`), а не через host-порт, поэтому смена
> host-bind его не затрагивает.

#### Шаг 2 — приватный туннель WireGuard (если ещё нет)

<details>
<summary>Быстрая настройка WireGuard бот↔сайт</summary>

**На сервере БОТА** (`10.8.0.1`, публичный IP — `BOT_PUBLIC_IP`):

```bash
apt install -y wireguard
wg genkey | tee /etc/wireguard/bot.key | wg pubkey > /etc/wireguard/bot.pub
# сгенерируй пару и для сайта (или сделай это на сайт-сервере):
wg genkey | tee /etc/wireguard/site.key | wg pubkey > /etc/wireguard/site.pub

cat > /etc/wireguard/wg0.conf <<EOF
[Interface]
Address = 10.8.0.1/24
ListenPort = 51820
PrivateKey = $(cat /etc/wireguard/bot.key)
[Peer]
PublicKey = $(cat /etc/wireguard/site.pub)
AllowedIPs = 10.8.0.2/32
EOF
systemctl enable --now wg-quick@wg0
```

**На сайт-сервере** (`10.8.0.2`):

```bash
apt install -y wireguard
cat > /etc/wireguard/wg0.conf <<EOF
[Interface]
Address = 10.8.0.2/24
PrivateKey = <site.key с бота>
[Peer]
PublicKey = <bot.pub с бота>
Endpoint = BOT_PUBLIC_IP:51820
AllowedIPs = 10.8.0.1/32
PersistentKeepalive = 25
EOF
systemctl enable --now wg-quick@wg0
wg show && ping -c2 10.8.0.1      # должен пойти handshake и пинг
```

> У провайдера сервера бота открой входящий **UDP 51820**.

</details>

#### Шаг 3 — на сайт-сервере: поставить кабинет

```bash
./install.sh site
```

Установщик ставит **только кабинет** (nginx + React). Спросит **3 значения**:

| Поле | Пример | Зачем |
|------|--------|-------|
| Username бота | `My_VPN_Bot` | вход в кабинет через Telegram |
| Приватный адрес API бота | `10.8.0.1:5000` | куда проксировать `/api/` (WG-адрес бота) |
| URL кабинета | `https://cabinet.example.com` | публичный адрес кабинета |

Секреты на сайт-сервере **не нужны** — их держит бот. Установщик заодно
пред-проверит связь с ботом и предупредит, если туннель не поднят.

#### Шаг 4 — на сайт-сервере: reverse-proxy с TLS

Поставь домен кабинета на IP сайт-сервера и подними TLS-прокси на `127.0.0.1:5002`:

```caddy
cabinet.example.com {
    reverse_proxy 127.0.0.1:5002
}
```

> ⚠️ **Если на сайт-сервере 443 уже занят** (например, это VPN-нода Remnawave с
> xray на 443) — кабинет на чистый `443` не встанет. Варианты: выделить под сайт
> отдельную машину со свободным 443, либо повесить кабинет на нестандартный порт
> (`cabinet.example.com:8443`). **Но нестандартный порт ломает вход через Telegram
> и Mini App** (Telegram ждёт `443`); email-вход при этом работает. Для боевого
> использования держи сайт там, где `443` свободен.

## 🌐 Порты после установки

Сервисы слушают **только локально** — наружу их выводит reverse-proxy с TLS:

| Сервис  | Адрес            | Назначение                      | Где |
|---------|------------------|---------------------------------|-----|
| Бот/API | `127.0.0.1:5000` | вебхуки Telegram, публичный API | сервер бота |
| Кабинет | `127.0.0.1:5002` | веб-кабинет                     | сервер кабинета (А: тот же; Б: отдельный) |

## 🛠 Управление

```bash
# удобный алиас
DC="docker compose -f docker-compose.yml -f cabinet/docker-compose.cabinet.yml"

$DC ps                 # статус контейнеров
$DC logs -f            # логи всех сервисов
$DC logs -f remnashop  # логи только бота/API
$DC up -d              # применить изменения .env
$DC up -d --build      # пересобрать после обновления кода
$DC down               # остановить всё
```

## ⚙️ Конфигурация (.env)

Все переменные описаны в [`.env.example`](.env.example) по разделам:

| Раздел | Что внутри |
|--------|------------|
| **Обязательно** | токен бота, домен, Remnawave, username — вводятся вручную |
| **Секреты** | `APP_CRYPT_KEY`, `*_SECRET`, пароль БД — генерируются |
| **Дефолты** | `WEB_ENABLED`, `BOT_MINI_APP*` — разумные значения |
| **Email / SMTP** | необязательно; для регистрации по почте |
| **Добавлено RemnaShop** | блок, который дописывает `install.sh` |

После правки `.env` примените изменения: `docker compose … up -d`.

## 🔒 Безопасность

- `.env` с секретами **не коммитится** — он в [`.gitignore`](.gitignore).
- Секреты генерируются криптостойко (`openssl rand`).
- Сервисы доступны только на `127.0.0.1` — наружу через TLS-прокси.
- Админ-доступ — fail-closed: только при явном `is_admin` от бэкенда.

## 🎨 Кастомизация (название и цвета)

Оформление меняется **в самом кабинете**, без правки кода:

**Админка → Оформление** (`/admin/appearance`):

| Что | Где применяется |
|-----|-----------------|
| **Название сервиса** | шапка кабинета, экран входа, заголовок вкладки |
| **Акцентный цвет** | кнопки, ссылки, выделения, свечения |
| **Цвет фона** | базовый фон (оттенки поверхностей подбираются сами) |

Есть живой предпросмотр; после «Сохранить» изменения видны всем пользователям.

> **Где это хранится.** Настройки оформления лежат в `assets/branding.json`
> (том `assets`, переживает пересоздание контейнера). Цвет `null` = взять из
> темы. Значение по умолчанию для названия — `RemnaShop`.

Юридические тексты (оферта, правила) на странице «Информация» правятся в коде:
[`cabinet/src/pages/InfoPage.tsx`](cabinet/src/pages/InfoPage.tsx).

## 💬 Частые вопросы

**Бот ещё не установлен — что делать?**
Сначала настройте бота RemnaShop (`cp .env.example .env`, заполните раздел
«Обязательно»), затем запустите `./install.sh`.

**Кабинет не открывается у некоторых операторов.**
Кабинет не зависит от внешних доменов на старте. Если домен режут по DPI —
проверьте доступность с включённым VPN; возможно, нужно сменить домен/спрятать за CDN.

**Как обновиться?**
`git pull`, затем `./install.sh` (или `docker compose … up -d --build`).
